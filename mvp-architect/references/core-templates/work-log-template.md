# WORK_LOG.md Template

Generate `docs/WORK_LOG.md` during Phase C as a living doc — the verbose counterpart to the one-line **Work tracker** in `AGENTS.md` / `CLAUDE.md`. At project birth it carries only the header and the Phase C setup section; it then grows **one `## <work item>` section per work item**, written when that item closes (sdi-mode Step 8 housekeeping).

The Work tracker is the *index* (one line per item: status + pointers); `WORK_LOG.md` is the *narrative* (the long story that used to bloat the tracker's `Notes` cell). Splitting them keeps the fact sheet — which is loaded every session — lean.

Two things `WORK_LOG.md` is **not**:

- **Not loaded every session.** Unlike `AGENTS.md` / `CLAUDE.md`, this is history. Read it on demand when you need the full story of a past work item; it is not part of the standard reading order before a phase.
- **Not the canonical source of detail.** The authoritative record stays in the linked artifacts: `docs/IMPLEMENTATION_PLAN_*.md`, `docs/memory/YYYY-MM-DD.md`, `docs/DECISIONS.md`, `docs/KNOWN_ISSUES.md`. Each section consolidates the narrative and points into them; new detail goes to those artifacts, not only here.

## Per-section format (one section per work item, added at close)

When a work item's end-of-phase housekeeping (sdi-mode Step 8) closes the item, add a section like this and reduce the item's Work tracker row to a single line:

```markdown
## <work item slug or Phase N>

**Type:** feature | bugfix | maintenance | migration | refactor | perf | phase · **Status:** ✓ | ✓ Merged→main | ⏸ | ... · **Date:** YYYY-MM-DD
**Plan:** docs/IMPLEMENTATION_PLAN_<name>.md · **PR:** [#NN](url) *(if any)*

[The verbose narrative that used to live in the tracker's Notes cell: what was built, the checkpoints/rounds, adversarial-review outcomes, test-count deltas, and cross-references — DECISIONS #N, KI-NNN, commit SHAs for fixes, PR links. Consolidate and point into the canonical artifacts rather than re-deriving their content.]
```

Rules:

- **Match the tracker.** The section's `Type` / `Status` / `Date` must match the corresponding Work tracker row, so the index and the narrative never disagree. The slug heading must match the row's `Work item`.
- **Order.** Keep sections in the same order as the tracker rows (chronological); append the newest at the end.
- **Cross-reference, don't duplicate.** "DECISIONS #28–#31 written", "KI-049 Resolved (commit, date)", "PR #7 → test" — link/point, don't paste the entries' content.
- **Write at close, not continuously.** During active rounds, detail lives in round reports, `docs/memory/`, `DECISIONS.md`, and `KNOWN_ISSUES.md`. The `WORK_LOG.md` section is the consolidated story written once the item closes (or updated if the item reopens for a follow-up).

## Length discipline

~20–40 lines at birth (header + Phase C section). It grows one section per work item; a section is typically 5–30 lines depending on the item's size. There is no hard cap — this file is allowed to be long because it is the archive — but each section should consolidate, not transcribe. If a section is reproducing the full content of DECISIONS / KNOWN_ISSUES / a round report, cut it down to pointers.

## What goes here vs the other surfaces

| Surface | Holds | Loaded each session? |
|---|---|---|
| **Work tracker** (`AGENTS.md` / `CLAUDE.md`) | one-line index per item: status + pointers | yes (part of the fact sheet) |
| **`WORK_LOG.md`** | the verbose per-item narrative, written at close | no — read on demand |
| `DECISIONS.md` | atemporal "why we chose X" | skimmed at audit |
| `KNOWN_ISSUES.md` | append-only "what we know is wrong" | skimmed at audit |
| `docs/memory/YYYY-MM-DD.md` | datable "what happened today / what's next" | recent entries, yes |

`WORK_LOG.md` is per-item and consolidated-at-close; `docs/memory/` is per-day and written continuously. Don't conflate them: a daily memory entry says "omie-integration closed today, see WORK_LOG"; the WORK_LOG section is the durable consolidated story of that item.
